AWS Synthetics With Terraform. Problems?

2022-03-08 :: Maksym Postument :: 4 min read (649 words)
#AWS  #Devops  #Terraform 

Привіт!

Недавно я за допомогою terraform автоматизовував створення AWS Synthetics, і стикнувся з декількома проблемами та хочу поділитись як я їх вирішив.

locals {
  common_tags = {
    cost-center = "observability"
    label       = "canary"
    environment = "staging"
  }
}

data "archive_file" "login" {
  type       = "zip"
  source_dir = "${path.module}/functions/login/"
  output_path = "${path.module}/functions/login.zip"
}

resource "aws_synthetics_canary" "login" {
  name                 = "login_test"
  artifact_s3_location = "s3://my_canary_bucket/"
  execution_role_arn   = "role_arn"
  handler              = "index.handler"
  zip_file             = data.archive_file.login.output_path
  runtime_version      = "syn-nodejs-puppeteer-3.4"
  start_canary         = true

  schedule {
    expression = "rate(5 minutes)"
  }

  run_config {
    timeout_in_seconds = 300
  }

  tags = merge(
    local.common_tags
  )
}

Перша проблема з якою я стикнувся те що Lambda яку автоматично створює canary не має тегів. Щоб її протегати я використав null_resource. Arn лямбди можна отримати за допомогою aws_synthetics_canary.login.engine_arn, але проблема в тому що arn містить версію лямбди arn:aws:lambda:us-east-1:123456789:function:my_lambda:15. А для того, щоб могти протегувати версія не потрібна. Щоб з лямбди забрати версію я додав наступне в locals.

  lambda_regex       = "arn:aws:lambda:[a-z]{2}-[a-z]+-\\d{1}:\\d{12}:function:[a-zA-Z0-9-_]+"
  login_lambda_name  = regex(local.lambda_regex, aws_synthetics_canary.login.engine_arn)

Тепер коли є Arn за допомогою null_resource можна протегувати лямбду. Triggers вказує в яких випадках запускати null_resource. В цьому випадку null_resource буде запускатись якщо поміняється тег в мапі common_tags або ж поміняється aws_synthetics_canary id (коли canary буде перестворено). І в provisioner "local-exec" запускаю aws lambda tag-resource в який передаю arn лямбди і теги. Єдине що для aws lambda теги мають бути в правильному форматі, а в terraform вони в вигляді мапи. Щоб їх конвертувати в тип string потрібно наступне додати в locals

  tags               = trimsuffix(join(" ", formatlist("%s=%s,", keys(local.common_tags), values(local.common_tags))), ",")

Таким чином після створення aws_synthetics_canary буде протеговано і лямбду.

resource "null_resource" "tag_login_backend_lambda" {
  triggers = {
    tags   = local.tags
    canary = aws_synthetics_canary.login.id
  }
  provisioner "local-exec" {
    command = "aws lambda tag-resource --resource ${local.login_lambda_name} --tags '${local.tags}'"
    environment = {
      AWS_PROFILE = "production
    }
  }

  depends_on = [aws_synthetics_canary.login]
}

Наступна проблема полягає в тому що aws_synthetics_canary ресурс не підтримує змінних оточення. Додати їх можна також за допомогою null_resource. Для цього ж знову потрібно додати змінні оточення в locals. В моєму випадку це буде мапа в якій буде тільки одна змінна це app_url

  canary_variables = {
    "app_url" = "https://mpostument.com"
  }

Triggers виглядає схоже до попереднього прикладу, тільки canary_variables з мапи потрібно перетворити в string, оскільки triggers підтримує тільки string. Складніше виглядає виклик команди aws synthetics update-canary, оскільки окрім змін оточення потрібно передати інші параметри, якщо їх не передати, то aws cli поставить їх в дефолтне значення.

resource "null_resource" "add_environment_variables_login_canary" {
  triggers = {
    canary_variables = join(",", [for key, value in local.canary_variables : "${key}=${value}"])
    canary           = aws_synthetics_canary.login.id
  }
  provisioner "local-exec" {
    command = "aws synthetics update-canary --name ${aws_synthetics_canary.login.name} --run-config '${jsonencode({ TimeoutInSeconds : aws_synthetics_canary.create.run_config[0].timeout_in_seconds, MemoryInMB : aws_synthetics_canary.create.run_config[0].memory_in_mb, ActiveTracing : aws_synthetics_canary.create.run_config[0].active_tracing, EnvironmentVariables : var.canary_variables })}'"
    environment = {
      AWS_PROFILE = var.aws_profile_name
    }
  }
  depends_on = [aws_synthetics_canary.create]
}

Ще одна проблема то що код canary автоматично не оновлюється, для того щоб він оновлювався назва архіву має бути іншою кожен раз. Це можна зробити за допомогою функції filemd5. Ось так виглядатиме archive_file з новими змінами

locals {
  login_function_location = "${path.module}/functions/login/nodejs/node_modules/index.js"
}

data "archive_file" "login" {
  type       = "zip"
  source_dir = "${path.module}/functions/login/"
  output_path = "${path.module}/functions/login-${filemd5(local.login_function_location)}.zip"
}

Таким чином кожного разу буде рахуватись хеш файлу і якщо він змінився то новий файл буде створено і canary код буде оновлено

Ще є одна проблема що при виклику terraform destroy не видаляється лямбда яка автоматично створюється aws_synthetics_canary. ЇЇ я поки що ще не вирішив.

UPD: В AWS провайдері версії 4.5.0 добавлена підтримка environment_variables. Для того щоб їх добавити потрібно в run_config вказати мапу з потрібними зміними оточення

locals {
  canary_variables = {
    "app_url" = "https://mpostument.com"
  }
  login_function_location = "${path.module}/functions/login/nodejs/node_modules/index.js"
}

data "archive_file" "login" {
  type       = "zip"
  source_dir = "${path.module}/functions/login/"
  output_path = "${path.module}/functions/login-${filemd5(local.login_function_location)}.zip"

resource "aws_synthetics_canary" "login" {
  name                 = "login_test"
  artifact_s3_location = "s3://my_canary_bucket/"
  execution_role_arn   = "role_arn"
  handler              = "index.handler"
  zip_file             = data.archive_file.login.output_path
  runtime_version      = "syn-nodejs-puppeteer-3.4"
  start_canary         = true

  schedule {
    expression = "rate(5 minutes)"
  }

  run_config {
    timeout_in_seconds    = 300
    environment_variables = local.canary_variables
  }

  tags = merge(
    local.common_tags
  )
}
