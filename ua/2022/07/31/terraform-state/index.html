В цьому файлі зберігається уся інформація про створений сервер. І якщо запустити terraform destroy то terraform піде в цей файл подивиться які ресурси були створено і видалить їх. Але що буде якщо видалити цей файл запустити знову terraform. В такому разі якщо запустити terraform apply то terraform не буде знати що такий сервер уже існує і створить його, і аналогічно з terraform destroy, terraform напише що сервер не існує і немає що видаляти.

$ terraform destroy

No changes. No objects need to be destroyed.

Either you have not created any objects yet or the existing objects were already deleted outside of Terraform.

Destroy complete! Resources: 0 destroyed.

Remote state⌗

А як ж тоді декілька розробників можуть працювати над одним кодом? В terraform є таке поняття як remote backend. Remote backend дозволяє зберігати state в remote storage як наприклад s3 і при кожному виклику terraform state файл буде використовуватись з remote storage і він зажди буде однаковим для будь кого хто викликає terraform і має відповідні права доступу. Але залишається ще одна проблема, якщо декілька розробників запустять terraform одночасно, чиї зміни тоді будуть в state файлі? Ця проблема вирішується ще одним механізмом який надає remote backend і який називається state lock. Але не всі backend підтримують state lock. В AWS для цього потрібно використати окрім s3 додатково ще dynamodb.

Спробуємо додати тепер remote backend. З попередньої статті provider.tf в нас виглядав так

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "4.23.0"
    }
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "default"
}

Для того щоб додати remote backend в terraform секцію додаткову конфігурацію і тоді файл буде виглядати так

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "4.23.0"
    }
  }
  backend "s3" {
    bucket = "BUCKET_NAME"
    key    = "tfstate/app1.tf"
    region = "us-east-1"
    profile = "default"
    encrypt = true
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "default"
}

Також рекомендується включити версійність в s3, для того щоб мати історію змін state файлу і могти вернутись до попередньої версії при потребі.

State Lock⌗

Для того щоб використовувати state lock в s3 потрібно створити dynamodb таблицю, якщо її створювати за допомогою terraform то її код виглядав б ось так

resource "aws_dynamodb_table" "state-lock" {
  name = "terraform-state-lock"
  hash_key = "LockID"
 
  attribute {
    name = "LockID"
    type = "S"
  }
 
  tags {
    Name = "Terraform state lock"
  }
}

І додати назву таблиці в terraform backend конфігурацію

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "4.23.0"
    }
  }
  backend "s3" {
    bucket = "BUCKET_NAME"
    key    = "tfstate/app1.tf"
    region = "us-east-1"
    profile = "default"
    encrypt = true
    dynamodb_table = "terraform-state-lock"
  }
}

Після цього тільки один запуск terraform буде працювати, якщо хтось запустить terraform в паралель то він всього лиш отримає повідомлення що state зараз є заблокованим.
