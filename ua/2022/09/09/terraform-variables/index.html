Привіт!

В цьому пості розкажу про параметри в terraform та terraform data.

Variables⌗

Використаю код з попередніх постів. Створю новий файл variables.tf і додам в ньому параметр instance_type.

variable "instance_type" {
  default = "t2.micro"
  description = "Instance type for test instance"
  type = string
}

В цьому блоці до параметра instance_type додано декілька атрибутів:
default - параметр буде мати значення вказане в цьому атрибуті якщо інше не буде вказано
description - опис параметра
type - тип параметра

Ці атрибути не є обов’язковимим. Параметр instance_type можна створити і так

variable "instance_type" {}

Але так складніше зрозуміти що це за параметр.
Після того як параметр створено його можна використати в terraform коді як var.instance_type

resource "aws_instance" "foo" {
  ami                    = "ami-0cff7528ff583bf9a"
  instance_type          = var.instance_type
  subnet_id              = "subnet-db73f0ac"
  vpc_security_group_ids = ["sg-b04b8cd4"]
  tags = {
    Env = "Dev"
  }
  volume_tags = {
    "Env" = "Dev"
  }
}

Locals⌗

Окрім звичайних параметрів в terraform можна також використовувати локальні. Їх зручно використовувати коли якесь значення часто повторюється але його немає потреби виносити в звичайні параметри. Створюються вони в блоці locals. Наприклад

locals {
  subnet_id = "subnet-db73f0ac"
}

І тепер цей локальний параметр можна використати в коді як local.subnet_id

resource "aws_instance" "foo" {
  ami                    = "ami-0cff7528ff583bf9a"
  instance_type          = var.instance_type
  subnet_id              = local.subnet_id
  vpc_security_group_ids = "sg-123"
  tags = {
    Env = "Dev"
  }
  volume_tags = {
    "Env" = "Dev"
  }
}

Data⌗

Деякі атрибути не завжди зручно використовувати як параметри, а краще динамічно отримувати значення при запуску terraform. Наприклад атрибут ami, якщо потрібно завжди мати актуальні версію ami не залежачи від регіону. Чи ж будь які інші ресурси які створенні іншим terraform кодом чи за допомогою AWS UI як наприклад VPC чи subnet_id. Які можуть змінитись і їх значення потрібно отримувати динамічно. Для цього в terraform є data.

Покажу як можна динамічно отримати значення ami. Для цього створю окремий файл який назву data.tf з таким контентом

data "aws_ami" "amzn" {
  most_recent = true
  owners = ["amazon"]

  filter {
    name = "name"
    values = ["amzn2-ami-kernel-*"]
  }
}

На початку блоку вказано data а не resource як раніше ми використовували. А в середині блоку декілька атрибутів:
most_recent - чи отримувати завжди останню версію чи ні
owners - який AWS account є власником ami
filter - по яких критеріях фільтрувати. В цьому випадку я фільтрую по імені яке має починатись з amzn2-ami-kernel. І таким чином я завжди буду мати ID останньої amazon linux 2 AMI не залежно від того в якому регіоні я буду створювати ресурси.

Тепер це можна використати в нашому aws_instance ресурсі як data.aws_ami.amzn.id

resource "aws_instance" "foo" {
  ami                    = data.aws_ami.amzn.id
  instance_type          = var.instance_type
  subnet_id              = local.subnet_id
  vpc_security_group_ids = "sg-123"
  tags = {
    Env = "Dev"
  }
  volume_tags = {
    "Env" = "Dev"
  }
}
